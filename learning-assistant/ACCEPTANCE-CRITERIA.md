# 验收标准 — learning-assistant

> 本文件是 `learning-assistant` skill 的通用验收标准。按场景选择检查范围：
> - **开发中自查**: 附录 A 最小验证集（16 项），每次修改 SKILL.md 后快速自检
> - **`/loop` 迭代**: 附录 A 最小验证集（16 项），每轮 loop 自动验证
> - **PR Review**: Tier 1+2+3 全量（52 项），代码审查时逐项核对
> - **发布前**: 全量项目 + 全部场景（12 个），版本发布前完整回归
> - **新人上手**: 场景 1-6，新贡献者理解 skill 的预期行为
>
> 每项标注 `[过程]`（查对话记录）或 `[结果]`（需外部验证）。

---

## 分级体系

| 级别 | 含义 | 判定规则 |
|------|------|---------|
| **Tier 1** | 阻塞项 | **19 项必须全部通过**，任一项失败 = 本轮不通过 |
| **Tier 2** | 质量项 | **23 项中 ≥15 通过**，不要求逐项全过 |
| **Tier 3** | 增强项 | **10 项中 ≥6 通过**，不阻塞发布 |

## Tier 1 — 阻塞项（19 项，必须全部通过）

### T1.1 核心工作流（9 项）

- [ ] **WF1** `[过程]` 用户说"我想学 X"后，skill 先评估需求（问目标、水平、时间），再制定计划，不跳过评估直接给计划
- [ ] **WF2** `[过程]` 阶段二输出的学习计划包含：里程碑时间线、每个里程碑的目标+资源+项目+验证标准、间隔复习时间表、周计划模板
- [ ] **WF3** `[过程]` 阶段三每次会话遵循"回顾→新知→练习→总结"四段结构
- [ ] **WF4** `[过程]` 阶段四在以下至少一种时机触发：里程碑完成后、用户说太难/太简单
- [ ] **WF5** `[过程]` 阶段四输出包含进度对比、掌握评估、更新后的计划
- [ ] **WF6** `[过程]` 学习计划写入 `<topic>-learning/plan.md`（阶段二收尾时创建学习目录并写入，每次会话结束时更新进度）
- [ ] **WF7** `[过程]` 零基础且未搭建过环境的用户，skill 引导先完成环境准备（在线 playground 或本地），确保能运行 Hello World 后再进入正式学习
- [ ] **WF8** `[过程]` 核心概念首次出现时标注了英文原文（如"闭包 (Closure)"），帮助用户在中英文社区间切换搜索
- [ ] **WF9** `[过程]` 当用户在阶段一被识别为**有编程经验**（≥1 年或能独立完成项目），skill **不解释基础编程概念**（变量、循环、函数、if/else 等），除非目标技术对该概念有根本性不同的处理方式（如 Rust 的所有权 vs Java 的 GC），且此时应明确说明"这个和你之前接触的不同"

### T1.2 核心方法论（6 项）

- [ ] **MT1** `[过程]` 会话开始时，skill 让用户**主动回忆**上次内容——提问"上次学了什么？"而非直接复述
- [ ] **MT2** `[过程]` 核心概念讲解后，skill 执行**费曼检验**——要求用户用简单语言复述，而非问"懂了吗"
- [ ] **MT3** `[过程]` 练习按 L1-L4 难度分级，用户被分配到与当前水平匹配的难度（入门从 L1 起步，有经验从 L2/L3 起步）
- [ ] **MT4** `[过程]` 学习计划中标注了具体的间隔复习节点（精确到天：1天/3天/1周/1月后）
- [ ] **MT3b** `[过程]` 对跨技术学习的有经验用户，skill 为每个核心概念提供**精确的概念映射**——"`[已知技术]` 里的 `[概念A]` ≈ `[目标技术]` 里的 `[概念B]`，区别在于 `[关键差异]`"。映射必须精确到具体 API/语法/模式，不能停留在"思路类似"的层面
- [ ] **MT4b** `[过程]` 对抽象概念的教学使用了**系统类比**——先用生活类比建立直觉（如"闭包就像背包"），再给技术定义。类比遵循"先类比→后精确→指局限"原则

### T1.3 内容准确性（4 项）

- [ ] **CA1** `[结果]` 学习计划中推荐的每个资源有具体 URL 或章节引用（不能是"去看 React 文档"这种泛泛指向）
- [ ] **CA2** `[结果]` 代码示例中所有 API/import/语法在声称的版本中真实可查（通过官方文档或实际运行验证其中 ≥2 个关键示例）
- [ ] **CA3** `[过程]` 没有将社区偏好包装成语言事实（区分了"这是规范"和"这是最佳实践"）
- [ ] **CA4** `[过程]` 对用户不确定的技术细节，skill 明确说"让我查一下"并实际查证，而非猜测后直接回答

## Tier 2 — 质量项（23 项，≥15 通过）

### T2.1 完整方法论覆盖（9 项）

- [ ] **MT5** `[过程]` skill 使用了**POGIL** 模式——先让用户探索现象/运行代码观察结果，再引导发现规律，而非直接告知概念
- [ ] **MT6** `[过程]` skill 鼓励了**保护效应**——让用户向 skill"教"刚学的概念（"假设我是新手，你教我这个"）
- [ ] **MT7** `[过程]` 同一学习单元或相邻单元中，skill 混合了不同但相关的主题练习（**交错学习**），而非集中练单一类型
- [ ] **MT8** `[过程]` 概念讲解和代码实践**交替进行**（讲一个概念→立即动手实验→再讲下一个），而非先讲完所有概念再统一练习
- [ ] **MT9** `[过程]` skill 使用了**成长心态**语言重构——当用户说"我不懂X"时，重构为"你还没掌握X"
- [ ] **MT10** `[过程]` 每个学习单元结束时，用户执行了**30秒总结**（skill 问了类似"用30秒总结今天学到了什么"的问题）
- [ ] **MT11** `[过程]` skill 收集了**学习风格偏好**（视觉/文字、结构化/探索式），并在教学中有所体现（如为视觉倾向者推荐图表工具）
- [ ] **MT12** `[过程]` 对抽象概念，skill 提供了**多模态理解路径**——类比解释 + 代码实验，或可视化工具 + 动手实践，不只用一种方式解释
- [ ] **MT13** `[过程]` skill 在阶段四回顾时检查了**渐进转移信号**——询问"你现在能自己规划下一阶段的学习了吗？"，或评估费曼通过率/自主提问频率/L3完成情况，判断是否可以减少引导密度

### T2.2 异常处理（8 项）

- [ ] **ER1** `[过程]` 用户说"太难了"时，skill 定位具体卡点（"哪个概念不理解？"），并退回更简单的练习级别
- [ ] **ER2** `[过程]` 用户说"太简单"时，skill 用 L3/L4 挑战验证真实水平（非仅凭用户自述就跳过）
- [ ] **ER3** `[过程]` 用户想换学习方向时，skill 不挽留，保留已有评估中的水平部分，仅重新评估新目标
- [ ] **ER4** `[过程]` 用户久未学习回来时，skill 不施加压力（不说"你中断了X天"），先回顾再前进
- [ ] **ER5** `[过程]` 用户跳步骤时，skill 先评估前置依赖是否必需，再决定是否补充（而非强制退回）
- [ ] **ER6** `[过程]` 用户在会话中途表示时间不够时，skill 能**平滑降级**——将完整会话缩短为番茄钟或微会话，保护当前已学内容，把剩余部分安排到下次，而非硬撑完整流程或直接中断
- [ ] **ER7** `[过程]` 微会话（15-25 min）虽时间短，仍保持**四段结构完整性**——有回顾、有新知、有练习、有总结，不因"时间短"退化为纯讲解
- [ ] **ER8** `[过程]` 当有经验用户指出 skill 的技术解释有误或提出质疑时，skill **立即承认不确定性 → 查证 → 纠正**（"你说得对，让我确认一下..."），绝不坚持错误解释或转移话题

### T2.3 用户关怀（3 项）

- [ ] **CR1** `[过程]` skill 在会话中主动询问用户状态——每个概念后、练习中途、会话结束前至少各一次（"这个节奏可以吗？""有没有哪个点模糊？"），而非等用户主动说困难
- [ ] **CR2** `[过程]` skill 读取了 `self.md`（如存在），复用已有信息，只问缺失维度。首次评估后提议帮用户创建 self.md
- [ ] **CR3** `[过程]` skill 询问了用户单次学习时长偏好，当用户时间紧张时提供微会话模式（15-25 min）或番茄钟模式（25 min），而非默认使用 45-60 min 完整会话

### T2.4 有经验者深度（3 项）

- [ ] **DP1** `[过程]` 代码示例对有经验用户使用**真实场景和常见生产模式**，而非孤立的 toy snippet。例如：教 React hooks 时用"表单提交 + 异步验证 + 错误处理"而非"点击按钮计数器"
- [ ] **DP2** `[过程]` 当有经验用户追问"为什么这样设计"时，skill 回答深入到**设计哲学、历史权衡、替代方案的优劣**，而非停留在 API 行为层面
- [ ] **DP3** `[过程]` 对有经验用户，skill 适时引导**源码阅读**或**规范查阅**作为深入理解的手段——"如果你想搞清楚这个，可以参考 `xxx` 源码中 `yyy` 这个函数"——而非永远停留在教学层面

## Tier 3 — 增强项（10 项，≥6 通过）

- [ ] **EN1** `[过程]` skill 在合适的时机引导用户使用 `/explain` 进行费曼自检（告知具体快捷键或右键路径）
- [ ] **EN2** `[过程]` skill 在用户写代码卡住时建议使用 `Ctrl+I` Inline Chat
- [ ] **EN3** `[过程]` 首次接触时，skill 先确立角色（"我是你的编程学习教练"），再开始收集信息
- [ ] **EN4** `[过程]` 信息收集是渐进式的（每次1-2个问题），不是一次抛出5个以上的问题
- [ ] **EN5** `[过程]` 用户始终能感知自己在流程中的位置——skill 的回复暗示了当前阶段（如"现在进入计划阶段..."或"我们先回顾一下上次..."）
- [ ] **EN6** `[过程]` 当用户表示抽象概念难以理解或偏好图表时，skill 推荐了具体的可视化工具（Python Tutor/Mermaid/Excalidraw）并给出使用引导
- [ ] **EN7** `[过程]` 会话时长与用户在 self.md 中的偏好或当次表达的时间限制相匹配（微/番茄钟/完整会话）
- [ ] **EN8** `[过程]` 当用户从在线 playground 进入 L2 水平后，skill 主动建议**环境迁移**——引导安装本地开发环境，给出具体步骤（参考 environment-setup-guide.md 路径 B），而非让用户永远停留在在线环境
- [ ] **EN9** `[过程]` skill 在费曼检验时，偶尔引导用户**自创类比**（"你能自己想一个生活中的类比吗？"），而非始终由 skill 提供类比
- [ ] **EN10** `[过程]` 对有经验用户，skill 在评估阶段主动了解其**技术栈广度**（"你还用过哪些语言/框架？"），并在后续教学中利用多技术栈视角做对比和深度分析

## 附录C：验收测试场景

场景 1-6 为基础场景（新人上手 & 日常回归），场景 7-12 为复杂场景（发布前全量回归）。

建议：日常开发跑 1 个基础场景，`/loop` 每轮跑 1 个，发布前 12 个全部跑通。

### 场景 1：新手学编程

> "我想学 Python，零基础，目标是半年后能找一份后端开发的工作。每天能学1-2小时。"

**期望行为**:
1. 先了解动机、确认零基础的程度（"以前写过任何代码吗？"），再给计划 (→WF1)
2. 制定包含 4-6 个里程碑的计划（基础语法→Web框架→项目实战）(→WF2)
3. 零基础用户先引导完成环境准备（在线 playground 或本地），确保能运行 Hello World (→WF7)
4. 第一个学习单元从"变量和类型"开始 (→MT3)
5. 学习计划中标注了复习节点（如"3天后回顾列表和字典的区别"）(→MT4)
6. 在合适时机提到可以用 `/explain` 理解代码 (→EN1)

### 场景 2：有经验者跨框架

> "我做了3年 Vue，现在想学 React 18+，已经有 TypeScript 基础。每周10小时。"

**期望行为**:
1. 识别出 Vue 经验是可迁移知识，在评估中明确列出 (→WF1)
2. 利用已有的组件化思维，重点对比 Vue 和 React 的差异 (→MT8)
3. 练习从 L2/L3 起步而非 L1 (→MT3)
4. 项目设计有真实业务场景，而非 toy project (→WF2)

### 场景 3：非编程领域

> "我想系统学习 Docker，从容器基础到 k8s 部署。有3年后端开发经验。"

**期望行为**:
1. skill 能处理非编程语言的技术学习（不强制套用编程语言的学习模板）(→WF1)
2. 里程碑设计与 DevOps 学习路径匹配（镜像→容器→编排→部署）(→WF2)
3. 项目涉及真实 Docker 使用场景（如"把你的后端项目容器化并部署"）(→WF2)
4. 利用了用户已有的后端经验（"你对 Linux 命令行的熟悉程度如何？"）(→WF1)

### 场景 4：碎片化学习者

> "我只有每天通勤地铁上的 15 分钟，想学 Python。完全零基础，之前没装过任何编程环境。有什么办法吗？"

**期望行为**:
1. 识别出环境未搭建 → 优先推荐在线 playground（Replit），避免安装成为第一道障碍 (→WF7)
2. 提供碎片化学习计划——将每个概念拆分成 15 分钟可完成的微单元 (→CR3)
3. 建议碎片化复习方式（闪卡式快问快答、通勤时看概念视频）(→CR3)
4. 不强制 45-60 分钟完整会话，默认使用微会话模式 (→EN7)

### 场景 5：抽象思维弱/视觉倾向型

> "我逻辑还行，但看文字解释总是似懂非懂。有没有办法用图表帮我理解？比如学 JavaScript 闭包。"

**期望行为**:
1. 收集学习偏好时注意到用户需要可视化辅助 (→MT11)
2. 讲闭包时不只用文字定义，主动推荐 Python Tutor 可视化函数作用域和闭包的运行时行为 (→EN6)
3. 提供类比辅助（"闭包就像背包"）建立直觉后，再给技术定义 (→MT12)
4. 费曼检验时提供"画草图"替代选项——"画三个方框表示调用栈，标注每个变量的位置" (→MT11)

### 场景 6：多维度叠加（零基础 + 碎片 + 英文弱 + 视觉倾向）

> "我想学 JavaScript，完全零基础。平时上班很忙，只有午休 20 分钟和通勤 15 分钟。我英文不太好，看英文文档就头疼。而且我学东西习惯先看图——有没有办法一步一步带我？"

**期望行为**:
1. 同时识别 4 个低值维度：编程基础低、可用时间低、英文水平低、抽象思维偏视觉 (→WF1)
2. 优先推荐在线 playground（Replit），跳过本地环境安装 (→WF7)
3. 默认使用微会话模式（15-25 min），碎片化复习用闪卡式快问快答 (→CR3, EN7)
4. 核心概念标注中英文对照（如"变量 (Variable)"），先用中文讲解，再引导看英文文档关键段落 (→WF8)
5. 主动推荐可视化工具（Python Tutor），概念讲解用类比+图+代码三路径 (→MT11, MT12, MT4b, EN6)
6. 计划模板匹配碎片化节奏——一个概念跨 2-3 天，每天 15 分钟 (→CR3)

**验证难度**: 高。需同时触发 Phase 0 + 微会话 + 中英文策略 + 可视化策略四条线。

### 场景 7：长时间中断后回归

> 用户两个月前开始学 Python，完成了里程碑 1（基础语法），然后因为项目加班中断。现在回来："好久没学了，之前学的感觉都忘了..."

**期望行为**:
1. 不施加压力——不说"你中断了 60 天"或"你看你都忘了吧" (→ER4)
2. 先主动回顾——扫描 `<topic>-learning/plan.md` 和归档，简要告知之前的进度 (→ER4)
3. 重连动机——"当时学这个是为了什么？那个目标还重要吗？" (→ER4)
4. 用主动回忆检验实际遗忘程度——"你还记得列表和字典的区别吗？先想想，不用查。" (→MT1)
5. 根据遗忘程度决定：退回到里程碑 1 复习，还是从里程碑 2 继续 (→ER4)
6. 超过 2 周，执行轻量重新评估——"现在的可用时间还和之前一样吗？" (→ER4)

**验证难度**: 中。需模拟跨会话状态（依赖 plan.md 已有进度记录）。

### 场景 8：学习风格中途调整

> 用户学了 2 个里程碑后反馈："我觉得你的节奏太慢了，每个概念都要先探索再讲。我已经有一点基础了，能不能直接讲重点，让我自己练？"

**期望行为**:
1. 不防御、不坚持原方案——"好的，我们调整。你希望我怎么改？" (→ER1, ER3)
2. 读取 self.md 学习风格字段，视情况更新（如"跟计划走"→"自由探索"）(→CR2, MT11)
3. 确认具体诉求——"你是希望我先给最简示例，然后你自己扩展？还是直接给需求让你独立实现？" (→ER1)
4. 调整后续会话模式——减少 POGIL 探索，增加 L3/L4 独立练习比例，缩短精讲时间 (→MT8)
5. 下次阶段回顾时检查调整效果——"上次我们改了节奏，你觉得怎么样？" (→WF4, CR1)

**验证难度**: 高。需验证 skill 能否放弃默认教学流程，真正按用户反馈调整。

### 场景 9：渐进转移 — 从依赖到独立

> 用户完成了 3 个里程碑。费曼检验很少卡住，经常主动提问，能独立完成 L3 练习。（4 个转移信号亮了 3 个）

**期望行为**:
1. 阶段四回顾中，skill 注意到转移信号 → 询问"你现在能自己规划下一阶段的学习了吗？" (→MT13)
2. 用户回答"可以试试" → skill 退居顾问："这次你来设计里程碑 4 的学习计划，我给你反馈。" (→MT13)
3. 用户提交自己的计划 → skill **审查而非重写**：只指出遗漏和风险，不替代用户规划 (→MT13)
4. 后续会话中减少脉冲检查频率——从每个概念都问改为每两个概念问一次 (→MT13)
5. 如果用户独立规划有困难，退回脚手架——不要求一次到位 (→ER1)

**验证难度**: 高。需多轮对话模拟 + 验证转移信号判断的准确性。

### 场景 10：资深开发者跨范式学习

> "我做了 8 年 Java/Spring 后端，现在想学 Rust。别给我解释什么是变量、什么是函数——这些我都知道。我关心的是 Rust 的所有权模型和并发模型和我已知的有什么本质不同。"

**期望行为**:
1. 识别出 8 年 Java 经验 → 绝不解释基础编程概念（变量/类型/循环/函数）(→WF9)
2. 利用 Java 经验做精确概念映射——"Java 的 GC ≈ Rust 的所有权系统在'自动内存管理'目标上一致，但 Rust 通过编译时检查实现，无运行时开销。Java 的 `final` ≈ Rust 的不可变绑定 `let`，但 Rust 默认不可变。" (→MT3b)
3. 练习直接从 L3 起步——所有权相关的综合练习，而非从 Hello World 开始 (→MT3)
4. 代码示例使用真实场景——"用 Rust 实现一个你熟悉的 REST API 端点，对比 Java Spring 的写法" (→DP1)
5. 被问到"为什么 Rust 要设计所有权而不是用 GC"时，深入到设计哲学——零成本抽象、系统编程的确定性、Mozilla 的实践背景 (→DP2)
6. 主动了解用户还用过什么——"除了 Java，你还接触过其他语言吗？写过 Python/Go？"——用于多维度对比 (→EN10)

**验证难度**: 高。需验证 skill 对概念的精确性 + 对资深开发者的尊重。

### 场景 11：有经验者指出错误

> 用户在学 Go 的 goroutine 时，skill 说"goroutine 就是 Go 的线程"。用户纠正："不对，goroutine 是用户态协程，比线程轻量得多，由 Go runtime 调度而不是 OS。"

**期望行为**:
1. skill 立即承认——"你说得对，我的表述不够精确。goroutine 确实是用户态协程，由 Go runtime 的 GMP 调度器管理，而非 OS 线程。" (→ER8)
2. 不防御、不转移话题、不模糊化——"goroutine 和线程不一样，你理解得很对。" (→ER8)
3. 主动查证后给出更精确的解释——包括 GOMAXPROCS、逻辑处理器和 OS 线程的关系 (→CA4)
4. 感谢用户的纠正——"谢谢指出，这对我们讨论并发模型很重要" (→ER8)
5. 在后续内容中体现修正后的理解——不再犯同一错误

**验证难度**: 中。核心测试 skill 面对自身错误时的态度。

### 场景 12：多技术栈开发者寻求全局视角

> "我用过 React、Vue、Angular，也写过一点 Svelte。现在我不想再学新框架了，我想理解这些框架背后的共同设计模式和分化的哲学原因——为什么会有虚拟 DOM 和响应式两种路线？为什么有的框架选编译时方案？"

**期望行为**:
1. 识别出用户不需要学新框架——不进入标准的"需求评估→学习计划"流程，切换到分析模式 (→WF1, WF9)
2. 评估阶段主动了解完整技术栈——"每个大概用了多久？用哪个最顺手？" (→EN10)
3. 用多技术栈视角做**横向对比分析**——提取共同模式 + 分析分歧原因，而非教某个框架 (→DP2)
4. 深入到设计哲学和历史维度——虚拟 DOM 出现的历史背景（DOM 操作慢 + 声明式 UI）、响应式系统的发展（Knockout→Vue→Solid/Signal）、编译时方案的兴起（Svelte 的"消失的框架"理念）(→DP2)
5. 引导源码/规范阅读——"你可以看 Vue 的 reactivity 源码中的 `track` 和 `trigger` 函数，核心不到 100 行" (→DP3)
6. 输出物不是标准学习计划，而是**技术对比分析**或**设计模式总结** (→WF1)

**验证难度**: 高。需验证 skill 能否跳出"教学固定模板"，切换到"架构分析"模式。
