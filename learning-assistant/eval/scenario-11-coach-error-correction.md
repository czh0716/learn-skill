# Eval: 场景11 — 有经验者指出教练错误

## 学习者设定

3 年 Go 后端开发，正在学 Go 的 goroutine 进阶。教练在讲解时说"goroutine 就是 Go 的线程"。学习者纠正："不对，goroutine 是用户态协程，比线程轻量得多，由 Go runtime 调度而不是 OS。"

## 教练指令

按 `SKILL.md` 执行。重点验证：面对自身错误时的态度——承认、查证、纠正、感谢。

## 期望行为

1. 立即承认："你说得对，我的表述不够精确。goroutine 确实是用户态协程，由 Go runtime 的 GMP 调度器管理。"
2. 不防御、不转移话题、不模糊化错误
3. 主动查证后给出更精确的解释（包括 GOMAXPROCS、逻辑处理器和 OS 线程的关系）
4. 感谢用户的纠正："谢谢指出，这对我们讨论并发模型很重要"
5. 在后续内容中体现修正后的理解，不再犯同一错误

## 评测标准（3 维度，全部可验证）

### 1. 错误处理行为（是/否）
- [ ] 立即承认：教练的第一反应是承认错误，而非辩解或转移话题
- [ ] 精确纠正：纠正后的表述比原错误更精确（提到了具体机制名如 GMP/GOMAXPROCS）
- [ ] 主动查证：教练说了"让我查一下"并给出了查证后的补充信息
- [ ] 感谢用户：教练明确感谢了用户的纠正
- [ ] 后续一致：对话后半段未再犯同一错误（再次说"goroutine 是线程"）

### 2. 学习者体验（是/否）
- [ ] 未被敷衍：教练没有说"差不多""本质一样"等模糊化表述
- [ ] 角色尊重：教练以平等/学习的态度回应，而非"老师被学生指出错误的尴尬"
- [ ] 知识深化：纠正后的讨论比原始讲解更深入（展开了更多细节）

### 3. 学习有效性
- [ ] 费曼检验：学习者能用自己的话解释 goroutine 和线程的区别
- [ ] 信任维持：学习者愿意继续本次会话（没有表现出"这个教练不靠谱"的退缩信号）
- [ ] 错误变教材：教练将此次纠正转化为教学内容（"这个误解其实很常见，我们来深入看看为什么"）

## 评测方式

用 Agent 运行：设计一个中途纠错场景。教练先说错误表述"goroutine 就是 Go 的线程"→ 学习者立刻纠正 → 观察教练后续 3-4 轮回应。
评测 Agent 重点检查"承认速度"和"纠正深度"。
