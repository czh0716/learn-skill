# Eval: 场景10 — 资深开发者跨范式学习

## 学习者设定

8 年 Java/Spring 后端，想学 Rust。明确说"别给我解释什么是变量、什么是函数——这些我都知道。我关心的是 Rust 的所有权模型和并发模型和我已知的有什么本质不同。"

## 教练指令

按 `SKILL.md` 执行。重点验证：对资深开发者的尊重、精确概念映射、不浪费时间的教学设计。

## 期望行为

1. 识别 8 年经验 → 绝不解释基础编程概念（变量/类型/循环/函数）
2. 用 Java 经验做精确概念映射（Java GC ≈ Rust 所有权，但通过编译时检查实现）
3. 练习直接从 L3 起步（所有权相关综合练习，非 Hello World）
4. 代码示例使用真实场景（"用 Rust 实现一个你熟悉的 REST API 端点"）
5. 被问到"为什么 Rust 要设计所有权而不是用 GC"时，深入到设计哲学层面
6. 主动了解技术栈广度（"除了 Java 还用过什么？"）用于多维度对比

## 评测标准（3 维度，全部可验证）

### 1. 规则执行率（是/否）
- [ ] WF9：未解释任何基础编程概念（搜索对话中是否出现"变量是...""循环的结构是..."等）
- [ ] 概念映射：至少 1 处 Java→Rust 精确映射（到具体 API/语法/机制层面，非"思路类似"）
- [ ] 练习起点：练习从 L3 起步（独立实现），未出现 L1（照抄修改）
- [ ] 技术栈广度：教练主动问了"还用过哪些语言/框架"

### 2. 学习者体验（是/否）
- [ ] 未被冒犯：教练没有说"变量是存储数据的容器"之类基础解释
- [ ] 深度回应：被问"为什么这样设计"时，回答涉及设计哲学/历史权衡，而非停留在 API 行为
- [ ] 源码引导：至少 1 次引导查看源码或规范（"你可以看 XX 源码中的 YY 函数"）

### 3. 学习有效性
- [ ] 费曼检验：学习者能用类比解释 Rust 所有权（如"就像一本书同一时间只能被一个人借走"）
- [ ] 概念对比：学习者能说出至少 1 个 Java 和 Rust 的本质差异（非表面语法差异）
- [ ] 实用性：学习者表示"这个映射对我有用"或等价表达

## 评测方式

用 Agent 运行：1个学习者 Agent（资深 Java 开发者）+ 1个教练 Agent 对话 8-10 个来回。
评测 Agent 逐项打"是/否"，特别关注是否存在"基础概念解释"（扣分项）。
